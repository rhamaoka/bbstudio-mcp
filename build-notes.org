* Project Goal
The goal of this project is to build a standalone Model Context Protocol (MCP) server that connects the Bluebeam Studio API with an Atlassian Forge Rovo Agent. This integration will empower users to extract and review document comments from Bluebeam sessions using simple, natural language queries directly within their Atlassian workspace. By centralizing the authentication and API logic, the MCP server acts as a secure, reusable bridge that seamlessly securely delivers your design data to enterprise AI tools.
* Plain Language Steps
The primary goal of this project is to design and deploy a standalone Model Context Protocol server that acts as a secure intermediary between AI clients, such as an Atlassian Rovo Agent, and the Bluebeam Studio API. This architecture will standardize the extraction of document comments and markups from Bluebeam sessions, enabling users to seamlessly query and retrieve project feedback using natural language directly from their connected workspaces. The resulting server will be reusable, modular, and responsible for handling all complex API interactions and authentication flows with Bluebeam.

To begin this project, you will first need to establish the necessary credentials with Bluebeam. This involves registering a new application within the Bluebeam Developer Portal to obtain your client ID and client secret, while ensuring you request the appropriate permissions for reading session data, document metadata, and markups.

Once your credentials are in place, the next phase is developing the core server application. You can choose a language like Node.js or Python and utilize the official Model Context Protocol SDKs to initialize a basic server structure. Within this server code, you will define the specific tools that the AI model can call. These tools will map to the Bluebeam API and include functions for listing available sessions, retrieving the documents within a specific session, and ultimately extracting the comments from a given document.

As you build these tools, you must implement a robust authentication flow. Your server needs to manage the OAuth 2.0 process to securely authenticate users with Bluebeam, handle the token exchange, refresh expired tokens, and attach the resulting access tokens to all outgoing requests made to the Bluebeam Studio API.

After local development and testing are complete, you will move to the deployment phase. You will need to host your server application on a reliable cloud platform that supports secure connections, ensuring it is accessible over the internet to your Atlassian environment.

The final stage involves configuring your Atlassian Rovo Agent to recognize and communicate with your newly deployed server. You will configure the agent within the Atlassian Forge environment to point to your server's endpoint, allowing it to automatically discover the Bluebeam tools you defined. From there, you can refine the agent's natural language instructions to ensure it understands exactly how to utilize the server to fetch session comments and how to format that information for the user.

** Design Decision 2: MCP Transport Protocol
Another critical decision is determining the specific MCP transport protocol your architecture will utilize. Because you are connecting a cloud-based client like an Atlassian Rovo Agent to a standalone server, you must use HTTP with Server-Sent Events rather than the standard input and output transport method typically used for local desktop applications. This decision dictates that your server application must include a web framework capable of handling incoming HTTP POST requests for tool execution and maintaining persistent Server-Sent Event connections to stream responses back to Rovo.

The Model Context Protocol transport layer is responsible for standardizing how messages are passed back and forth between an AI client and your data server. At its core, the protocol uses a standard messaging format called JSON-RPC 2.0, which ensures that every request for data, execution of a tool, or error message follows a predictable, structured layout. To actually move these JSON messages between systems, the protocol relies on two primary transport mechanisms: standard input and output for local connections, and HTTP with Server-Sent Events for remote connections.

The local transport method, utilizing standard input and output streams, is typically used when the AI application and the server are running on the exact same machine. In this scenario, the AI client launches the server as a background process and communicates with it directly through the operating system's basic data streams. While this is highly efficient for desktop AI assistants, it is not applicable to your project because the Atlassian Rovo agent lives in the cloud and cannot directly launch local processes on your machine.

For cloud-based integrations like yours, the protocol uses the HTTP transport method, which relies heavily on a web technology called Server-Sent Events. Server-Sent Events allow a web server to open a single, long-lived, one-way connection to a client. When the Atlassian Rovo agent connects to your Express server, your server will establish this persistent stream. This open channel is how your server will push all notifications, tool results, and responses back to the Rovo agent as soon as they are ready, without the agent needing to constantly ping the server for updates.

Because Server-Sent Events only allow data to flow from the server to the client, the protocol uses standard HTTP POST requests for the other half of the conversation. When the Rovo agent needs your server to execute a tool, such as extracting comments from a Bluebeam document, it packages that request into a JSON message and sends it via an HTTP POST request to a specific web address on your Express server.

Your Express server will be designed to listen for these incoming POST requests. Once it receives one, it will execute the required Bluebeam API calls, format the extracted comments, and then push the final JSON response back down the open Server-Sent Events stream it previously established with the agent. This combination of standard HTTP POST requests for incoming commands and Server-Sent Events for outgoing streaming responses allows two completely separate cloud environments to communicate securely and reliably in real time.

** Design Decision 3: Hosting and Deployment
You must also decide on the hosting and deployment infrastructure for the MCP server. You can choose a containerized approach using Docker deployed on fully managed serverless container platforms like Google Cloud Run or AWS Fargate, or opt for a traditional virtual machine. Containerization is highly recommended for MCP servers as it ensures consistent environments between local development and production. This decision will define which specific cloud provider accounts you need to create and what container registry and deployment pipelines are required.

Because the Atlassian Rovo Agent operates entirely in the cloud, your Node.js and Express server must be deployed to the public internet and secured with an HTTPS connection. You have a few primary paths for hosting this application, ranging from very beginner-friendly to enterprise-grade scalable solutions. The main options include Platform-as-a-Service providers, serverless container platforms, and traditional virtual machines.

The easiest approach is using a Platform-as-a-Service, with popular choices being Render, Heroku, or Railway. These platforms are incredibly beginner-friendly because they handle almost all the infrastructure work for you. You typically just connect your GitHub repository to the platform, and every time you update your code, the platform automatically builds your Node.js application, assigns it a secure HTTPS web address, and keeps it running. This is often the best choice if you want to get up and running quickly without learning about underlying server operations, and many of these providers offer inexpensive or free tiers for development.

The second option is using serverless container platforms like Google Cloud Run, AWS App Runner, or Azure Container Apps. For this method, you package your Node.js application and all its dependencies into a standardized unit called a Docker container. You then upload this container to your cloud provider. These serverless platforms are excellent because they automatically scale up when your Rovo Agent sends many requests and scale down to zero when not in use, meaning you only pay for the exact compute time you consume. While this is the industry standard for production deployments due to its security and scalability, it does require you to learn the basics of Docker.

The third option is a traditional virtual machine, such as an AWS EC2 instance or a DigitalOcean Droplet. This gives you a blank server where you manually install Node.js, configure the web server, set up firewalls, and manage your own SSL certificates for HTTPS. While it offers total control, it requires significant maintenance and is generally considered overkill for a lightweight Model Context Protocol server.

For the core technology and hardware, the project will be built using Node.js and the Express web framework, running directly on the operating system of your Raspberry Pi 5. This bare-metal approach avoids the extra complexity of Docker containers while taking advantage of your existing hardware. To ensure the application runs continuously in the background and automatically restarts if the device ever reboots, you will use the PM2 process manager to oversee the Node.js execution.

To connect this local server to the cloud-based Atlassian Rovo agent, the system will leverage your existing Cloudflare Tunnel setup. You will create a dedicated subdomain on your current domain to securely route external internet traffic past your home network's firewall. Once the traffic reaches the Raspberry Pi, your existing Nginx web server will act as a reverse proxy. Nginx will take the incoming requests from Cloudflare and quietly forward them to the specific local port where your Express application is listening.

Because the Model Context Protocol requires a constant stream of data for cloud connections, your Express server will communicate using HTTP with Server-Sent Events. To ensure this continuous stream works properly, you will need to adjust the Nginx configuration for this specific subdomain to disable proxy buffering and read timeouts. This simple configuration tweak ensures that Nginx instantly passes the data from your Express server back to the Atlassian Rovo agent without trying to hold onto the messages, allowing for seamless, real-time communication between your home server and the Atlassian cloud.

** Design Decision 4: Authentication and Secrets Management
The authentication strategy for the Bluebeam Studio API is another vital design choice that impacts your database needs. You need to determine whether the integration will use a machine-to-machine service account for all requests using the client credentials flow, or if it requires individual user authentication using the authorization code flow. If you require user-level authentication to ensure users only see comments they have permission to view in Bluebeam, you will need to design a secure token storage system. This requires provisioning a managed database like PostgreSQL or Redis to handle the OAuth token lifecycle, securely mapping Atlassian user IDs to their respective Bluebeam access and refresh tokens.

Your decision to use a single, shared account for bb-dev@lbl.gov drastically simplifies the entire architecture of your project. By bypassing individual user logins, you eliminate the need to build complex web interfaces for OAuth pop-ups, and you completely remove the need to securely map Atlassian user IDs to individual Bluebeam accounts.

Because you only need to manage one single set of credentials, you do not need to install or configure a database like SQLite. Instead, your Node.js application can handle token storage using a simple, secure local text file, such as a JSON file, stored directly in your project directory on the Raspberry Pi.

To make this work securely with Bluebeam's system, you will perform a one-time manual setup phase. You will generate an initial authorization code by logging in as bb-dev@lbl.gov through a web browser. You will provide this code to your Node.js server once, and the server will exchange it for a temporary Access Token and a long-lived Refresh Token, saving them to your local JSON file.

From that point forward, your server operates completely in the background. You will store your existing Client ID and Client Secret safely in a hidden .env text file on your Raspberry Pi, keeping them entirely out of your application's source code. Whenever the Atlassian Rovo agent requests document comments, your Express server will check if its current Bluebeam Access Token has expired. If it has, the server will automatically use the hidden Client credentials and the saved Refresh Token to silently fetch a new Access Token, update the local JSON file, and immediately proceed with fetching the requested data.

const fs = require('fs');
const axios = require('axios');
require('dotenv').config();

async function getValidToken() {
  let tokens = JSON.parse(fs.readFileSync('tokens.json', 'utf8'));

  if (Date.now() < tokens.expires_at) {
    return tokens.access_token;
  }

  // Token expired, refresh it
  const credentials = Buffer.from(
    `${process.env.BLUEBEAM_CLIENT_ID}:${process.env.BLUEBEAM_CLIENT_SECRET}`
  ).toString('base64');

  const body = new URLSearchParams({
    grant_type: 'refresh_token',
    refresh_token: tokens.refresh_token
  });

  const response = await axios.post(
    'https://authserver.bluebeam.com/auth/oauth/token',
    body.toString(),
    {
      headers: {
        'Authorization': `Basic ${credentials}`,
        'Content-Type': 'application/x-www-form-urlencoded'
      }
    }
  );

  const newTokens = response.data;
  newTokens.expires_at = Date.now() + (newTokens.expires_in - 60) * 1000;
  // Preserve refresh token if a new one was not issued
  if (!newTokens.refresh_token) newTokens.refresh_token = tokens.refresh_token;

  fs.writeFileSync('tokens.json', JSON.stringify(newTokens, null, 2));
  return newTokens.access_token;
}

The two HTTP endpoints serve fundamentally different purposes. The GET /mcp/sse endpoint opens a long-lived connection that stays open for the entire Claude conversation — the server pushes results to Claude through this stream. The POST /mcp/messages endpoint receives short-lived requests from Claude whenever it wants to invoke a tool. Both endpoints feed into the same transport instance, which is why a single transport variable is shared between them in the code.

The Model Context Protocol Software Development Kit handles all the protocol details for you. Your code only registers two handlers (one for listing tools, one for calling them) and the SDK takes care of the JSON-RPC framing, message routing, error handling, and Server-Sent Events formatting.

The authentication layer is intentionally isolated in its own module. Every tool invocation goes through getValidToken(), which means token refresh logic runs at most once per tool call and is centralized in one place. If Bluebeam ever changes its authentication scheme, you change one file.

The Bluebeam Studio Application Programming Interface is the only external dependency that actually executes business logic. Everything else in the diagram is either glue code or infrastructure. This is a good sign architecturally — it means the system is mostly thin plumbing around a well-defined external service.

const express = require('express');
const { randomuuid } = require('node:crypto');
const { server } = require('@modelcontextprotocol/sdk/server/index.js');
const {
  streamablehttpservertransport
} = require('@modelcontextprotocol/sdk/server/streamablehttp.js');
const { isinitializerequest } = require('@modelcontextprotocol/sdk/types.js');

const { registerhandlers } = require('./mcp-tools');

const app = express();
app.use(express.json());

// map of active sessions keyed by mcp-session-id header value.
// each entry holds the transport instance dedicated to that session.
const transports = {};

/**
 * handles all mcp requests on a single endpoint. the streamable http
 * transport multiplexes initialization, client-to-server messages,
 * and server-to-client streams over post and get to /mcp.
 */
app.post('/mcp', async (req, res) => {
  const sessionid = req.headers['mcp-session-id'];
  let transport;

  if (sessionid && transports[sessionid]) {
    // reuse existing transport for this session.
    transport = transports[sessionid];
  } else if (!sessionid && isinitializerequest(req.body)) {
    // new initialization request — create a fresh transport and server.
    transport = new streamablehttpservertransport({
      sessionidgenerator: () => randomuuid(),
      onsessioninitialized: (newsessionid) => {
        transports[newsessionid] = transport;
      }
    });

    // clean up when the transport closes.
    transport.onclose = () => {
      if (transport.sessionid) {
        delete transports[transport.sessionid];
      }
    };

    const server = new server(
      { name: 'bluebeam-mcp', version: '1.0.0' },
      { capabilities: { tools: {} } }
    );
    registerhandlers(server);
    await server.connect(transport);
  } else {
    // request lacks a valid session id and is not an initialization request.
    res.status(400).json({
      jsonrpc: '2.0',
      error: {
        code: -32000,
        message: 'bad request: no valid session id provided'
      },
      id: null
    });
    return;
  }

  await transport.handlerequest(req, res, req.body);
});

 * get and delete on /mcp are used by the transport for the server-to-client
 * stream and for session (mcp connection session, not bluebeam session) termination, respectively. both require an existing
 * session id.

  const sessionid = req.headers['mcp-session-id'];
  if (!sessionid || !transports[sessionid]) {
    res.status(400).send('invalid or missing session id');
    return;
  }
  const transport = transports[sessionid];
  await transport.handlerequest(req, res);
};

app.get('/mcp', handlesessionrequest);

/**
 * Builds a detailed, log-friendly error message from an axios error,
 * including the status code, the request URL, and the response body
 * when available. Falls back to a plain message for non-HTTP errors.
 */
const formatAxiosError = (e) => {
  let errorMessage = `Error: ${e.message}`;

  if (e.response) {
    errorMessage += `\nStatus: ${e.response.status} ${e.response.statusText || ''}`.trim();
    errorMessage += `\nURL: ${e.config?.method?.toUpperCase()} ${e.config?.url}`;
    if (e.response.data) {
      const body = typeof e.response.data === 'string'
        ? e.response.data
        : JSON.stringify(e.response.data, null, 2);
      errorMessage += `\nResponse body: ${body}`;
    }
  } else if (e.request) {
    errorMessage += `\nNo response received from ${e.config?.url}`;

 * Model Context Protocol Server instance. This is the single entry
 * point the transport layer uses to wire tools into a server.
 */
function registerHandlers(server) {
  server.setRequestHandler(ListToolsRequestSchema, async () => ({
    tools: TOOL_DEFINITIONS
  }));

  server.setRequestHandler(CallToolRequestSchema, async (request) => {
    const toolName = request.params.name;
    const args = request.params.arguments || {};
    const handler = TOOL_HANDLERS[toolName];

    if (!handler) {
      return {
        content: [{ type: 'text', text: `Unknown tool: ${toolName}` }],
        isError: true
      };
    }

      const token = await getValidToken();
      const headers = { Authorization: `Bearer ${token}` };
      const data = await handler(args, headers);
      return { content: [{ type: 'text', text: JSON.stringify(data) }] };

      console.error('[Bluebeam MCP] Tool call failed:', errorMessage);

        content: [{ type: 'text', text: errorMessage }],
        isError: true
      };
    }
  });

*** TODO Create a Docker Compose Configuration File
Create a file named `docker-compose.yml` in the project root. This file describes how the container should be run, including which ports to expose, which files to mount, and what restart policy to apply.
#+BEGIN_SRC yaml
services:
  bluebeam-mcp:
    build: .
    container_name: bluebeam-mcp
    restart: unless-stopped
    ports:
      - "3000:3000"
    volumes:
      - ./.env:/app/.env:ro
      - ./tokens.json:/app/tokens.json
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000/mcp/sse"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s

A few details worth understanding:
- The `:ro` suffix on the environment file mount means read-only. The application never writes to this file.
- The `tokens.json` mount is read-write because the token refresh logic updates the file when access tokens expire.
- The `restart: unless-stopped` policy replaces what Process Manager 2 was doing previously: the container restarts automatically if it crashes or if the Pi reboots, but stays stopped if you explicitly stop it.
- The healthcheck periodically verifies the server is responding. Docker marks the container as unhealthy if checks fail repeatedly, which is visible in `docker ps` output.

***** TODO Set Up Automated Image Builds Using GitHub Actions
If your repository is on GitHub, you can configure a workflow that automatically builds the Docker image whenever you push to the main branch and publishes it to a container registry such as GitHub Container Registry or Docker Hub. The Raspberry Pi would then pull the prebuilt image rather than building locally, which is faster and consumes fewer resources on the Pi.

Create `.github/workflows/docker-build.yml` in your repository:
#+BEGIN_SRC yaml
name: Build and Publish Docker Image

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/arm64
          push: true
          tags: ghcr.io/yourusername/bluebeam-mcp:latest
#+END_SRC
The `platforms: linux/arm64` line is important: it cross-compiles for the Raspberry Pi's ARM architecture rather than the x86_64 architecture of the GitHub Actions runner.

You would then change your `docker-compose.yml` on the Pi to use the prebuilt image rather than building locally:

***** TODO Add a Container Update Automation Tool
Watchtower is a small companion container that monitors your running containers and automatically pulls updated images and restarts the containers when new versions are published. This pairs naturally with the GitHub Actions workflow above.

Add Watchtower to your `docker-compose.yml`:
#+BEGIN_SRC yaml
  watchtower:
    image: containrrr/watchtower
    container_name: watchtower
    restart: unless-stopped
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - WATCHTOWER_POLL_INTERVAL=3600
      - WATCHTOWER_CLEANUP=true
    command: bluebeam-mcp
#+END_SRC
The `WATCHTOWER_POLL_INTERVAL=3600` setting checks for updates every hour. The `WATCHTOWER_CLEANUP=true` setting removes old images after a successful update to prevent disk space accumulation. The `command: bluebeam-mcp` argument restricts Watchtower to monitoring only that one container rather than every container on the host.

Note that automated updates carry some risk: if a new image is broken, your service goes down without warning. For a low-traffic internal service this is usually acceptable, but for anything more critical you may prefer manual updates so you can verify each release.
